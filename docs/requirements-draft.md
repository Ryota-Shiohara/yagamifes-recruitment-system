# 委員会新規局員採用システム 要件定義（確定版 v1.0）

## 1. この文書の位置づけ

本書は、レポート課題3で設計したデータベースを利用する最終課題Webアプリケーションの確定要件である。第12節の決定事項を前提として、スキーマと画面仕様を定める。

参照資料は次の3点である。

- 「レポート課題3」: 委員会新規局員採用システムの題材、ER図、スキーマ、検索要件
- 「データモデリング 第12回」: 最終課題の制約、評価ポイント、入力時の必須事項
- 「DBアプリプログラミング実習」: Flaskアプリの構成、デバッグ、デプロイ、提出方法

## 2. 授業上の必須条件

- Python 3.13.9、Flask 3.1.2、Python標準の `sqlite3` モジュール、SQLiteを使用する。
- SQLAlchemyなどのORM、SQLを書かずにDB操作を行うライブラリは使用しない。
- 本番環境で利用できない追加モジュールは使用しない。新しいモジュールが必要な場合は教員へ相談する。
- Webアプリは複数ページで構成する。
- SELECT文を使用し、データモデルを実際の画面・操作に活用する。
- ユーザー入力を受けるため、サーバー側バリデーション、エラー処理、セキュリティ対策を実装する。
- 可能な範囲で INSERT、UPDATE、DELETE、複数表の操作、集約関数、VIEW、トランザクションを使用し、レポートでSQLの意味を説明する。
- Webアプリには提出者の氏名・学籍番号を表示しない。
- 個人情報・権利上の問題を避けるため、公開環境では架空の応募者・面接官データのみを使用する。
- 最終提出は、レポートPDF、アプリ一式ZIP、本番URLの3点をそろえる。

## 3. システムの目的

応募者の応募情報と面接可能時間を登録し、局と時間の組で定まる面接枠へ応募者を割り当て、面接後の評価入力、順位確認、採否決定までを一つのリレーショナルデータベースで管理する。

この課題では、単に画面を作ることではなく、次を確認できることを重視する。

- 多対多関連を中間テーブルで表現できている。
- 主キー、候補キー、外部キー、UNIQUE制約、CHECK制約が業務ルールを表している。
- 複数表の結合と集約により、選考に必要な情報を検索できる。
- 複数表を更新する処理をトランザクションとして扱える。
- 全テーブルがBCNFであることを説明できる。

## 4. 対象範囲

### 4.1 実装する機能

1. 局、面接官、評価項目、共通時間帯、局別面接枠の参照
2. 応募者情報と複数の面接可能枠の一括登録
3. 応募者一覧の局・氏名・日程確定状況による検索
4. 応募者詳細と希望枠の確認
5. 希望枠の中から一つを選ぶ手動面接割り当て
6. 面接官による評価項目ごとの1〜10点評価の登録・修正
7. 評価者ごとの採点傾向を補正した局別ランキング、採用可能人数、採否の確認
8. 局別面接枠の追加・修正・削除

### 4.2 今回は実装しない機能

- メール送信、通知、リマインダー
- 全応募者を自動で最適配置するスケジューラー
- 実在する個人情報の管理
- ファイルアップロード
- 本格的なログイン、パスワード管理、アクセス制御
- 複数年度をまたぐ採用履歴

## 5. 利用者と権限の扱い

講義資料では最終課題に認証は不要とされているため、入口ページから次の役割別メニューへ遷移できる構成とする。

- 応募者メニュー: 応募情報と希望面接枠を登録する。
- 面接官メニュー: 担当面接と評価入力を確認する。
- 局責任者メニュー: 応募者、局別面接枠、割り当て、順位、採否を管理する。

これは役割別の操作を説明するための画面分離であり、セキュリティ上のアクセス制御ではない。実際の権限管理を要件に含める場合は、認証方式を別途定義しなければならない。

## 6. 用語

- 共通時間帯: 開始日時と終了日時だけを持つ、全局で再利用可能な時間区間。
- 局別面接枠: ある局が一つの共通時間帯を面接に利用可能としたもの。
- 希望面接枠: 応募者が参加可能として選択した局別面接枠。
- 確定面接枠: 応募者に最終的に一つ割り当てた局別面接枠。
- 評価項目: 局ごとに定める面接評価の観点。
- 評価: 一人の面接官が、一人の応募者を、一つの評価項目について付けた1〜10の整数値。

## 7. データモデル案

### 7.1 テーブル

#### bureaus（局）

- `id` INTEGER PRIMARY KEY
- `name` TEXT NOT NULL UNIQUE
- `capacity` INTEGER NOT NULL CHECK (`capacity >= 0`)

#### interviewers（面接官）

- `id` INTEGER PRIMARY KEY
- `name` TEXT NOT NULL
- `bureau_id` INTEGER NOT NULL REFERENCES `bureaus(id)`
- `role` TEXT NOT NULL CHECK (`role IN ('chief', 'interviewer')`)

#### criteria（評価項目）

- `id` INTEGER PRIMARY KEY
- `bureau_id` INTEGER NOT NULL REFERENCES `bureaus(id)`
- `name` TEXT NOT NULL
- `description` TEXT NOT NULL
- UNIQUE (`bureau_id`, `name`)

#### schedules（共通時間帯）

- `id` INTEGER PRIMARY KEY
- `start_at` DATETIME NOT NULL
- `end_at` DATETIME NOT NULL
- CHECK (`start_at < end_at`)
- UNIQUE (`start_at`, `end_at`)

日時は日本時間のISO 8601拡張形式 `YYYY-MM-DD HH:MM:SS` で保存する。

#### bureau_schedules（局別面接枠）

- `id` INTEGER PRIMARY KEY
- `bureau_id` INTEGER NOT NULL REFERENCES `bureaus(id)` ON DELETE CASCADE
- `schedule_id` INTEGER NOT NULL REFERENCES `schedules(id)` ON DELETE CASCADE
- UNIQUE (`bureau_id`, `schedule_id`)

人工キー `id` は、応募者希望、確定面接枠、URLから一つの局別面接枠を参照しやすくするために追加する。候補キー (`bureau_id`, `schedule_id`) もUNIQUE制約で保持する。

#### applicants（応募者）

- `id` INTEGER PRIMARY KEY
- `name` TEXT NOT NULL
- `email` TEXT NOT NULL UNIQUE
- `bureau_id` INTEGER NOT NULL REFERENCES `bureaus(id)`
- `remark` TEXT NOT NULL DEFAULT ''
- `confirmed_bureau_schedule_id` INTEGER UNIQUE REFERENCES `bureau_schedules(id)`
- `decision` TEXT CHECK (`decision IN ('accepted', 'rejected', 'waitlisted', 'withdrawn') OR decision IS NULL`)

`confirmed_bureau_schedule_id` のUNIQUE制約により、一つの局別面接枠へ二人を予約できないようにする。日程未確定時はNULLとする。採否未決定時の `decision` もNULLとする。辞退者は `withdrawn` とし、ランキング対象から除外する。

#### applicant_availabilities（応募者の希望面接枠）

- `applicant_id` INTEGER REFERENCES `applicants(id)` ON DELETE CASCADE
- `bureau_schedule_id` INTEGER REFERENCES `bureau_schedules(id)` ON DELETE CASCADE
- PRIMARY KEY (`applicant_id`, `bureau_schedule_id`)

応募者と局別面接枠の多対多関連を表す。レポート課題3で未実装だったが、対象概要に記載した業務を実現するため最終課題では必須とする。

#### scores（評価）

- `applicant_id` INTEGER REFERENCES `applicants(id)` ON DELETE CASCADE
- `interviewer_id` INTEGER REFERENCES `interviewers(id)` ON DELETE CASCADE
- `criterion_id` INTEGER REFERENCES `criteria(id)` ON DELETE CASCADE
- `score` INTEGER NOT NULL CHECK (`score BETWEEN 1 AND 10`)
- PRIMARY KEY (`applicant_id`, `interviewer_id`, `criterion_id`)

評価を一意に識別する候補キーそのものを主キーにする。レポート課題3の人工キー `scores.id` は、上書き可能性を高めるものではなく、候補キーに対する追加の識別子になっていたため削除する。

### 7.2 VIEW

`applicant_overview` を定義し、応募者、局、確定時間帯を左外部結合した一覧を返す。

- VIEW内には特定局に固定するWHERE句を入れない。
- VIEW内には不要なORDER BYを入れない。
- 局、氏名、日程確定状況の条件は、VIEWを使用するSELECT文側で指定する。
- 日程未確定の応募者も一覧から消えないようLEFT JOINを使用する。

### 7.3 アプリケーションとDBトリガーで検証する表横断ルール

次の制約は単純な外部キーだけでは表現しにくいため、アプリケーションで更新直前に検証する。さらにSQLiteトリガーでも検証し、別経路からSQLを実行した場合を含めて違反データを保存しない。

- 応募者が選べる希望枠は、応募先の局に属する局別面接枠だけである。
- 確定面接枠は、その応募者が事前に選択した希望枠の一つである。
- 確定面接枠は、応募先の局に属する。
- 同じ局の同じ共通時間帯には、局別面接枠を一つだけ作成できる。
- 同じ局に、時間の一部または全部が重なる複数の局別面接枠を作成できない。
- 一つの局別面接枠を確定面接枠として使用できる応募者は一人だけである。
- 評価者は応募者の応募先と同じ局に所属する面接官である。
- 評価項目は応募者の応募先局に属する。
- 開始日時が終了日時より前である。

### 7.4 評価者補正の定義

評価者による採点の甘辛を補正するため、評価者と評価項目の組ごとにz得点を計算する。同じ評価者でも評価項目によって採点傾向が異なる可能性があるため、評価者全体ではなく (`interviewer_id`, `criterion_id`) 単位で補正する。

応募者 `a`、評価者 `i`、評価項目 `c` の素点を `x(a,i,c)` とする。

1. `n(i,c) = COUNT(x)` として、評価者 `i` が評価項目 `c` で採点した応募者数を求める。
2. `μ(i,c) = AVG(x)` として、その評価者・評価項目の平均を求める。
3. `σ(i,c) = sqrt(AVG(x * x) - μ(i,c) * μ(i,c))` として母標準偏差を求める。
4. `σ(i,c) > 0` のとき、`z(a,i,c) = (x(a,i,c) - μ(i,c)) / σ(i,c)` とする。
5. 表示用の補正得点を `T(a,i,c) = 50 + 10 * z(a,i,c)` とする。
6. 応募者の補正後総合得点は、その応募者に対するすべての `T(a,i,c)` の算術平均とする。

例外は次のように扱う。

- `n(i,c) < 2` の場合、評価者の傾向を推定できないため、その評価者・評価項目は「補正用標本不足」とする。補正後順位には使用しない。
- `σ(i,c) = 0` の場合、その評価者が全応募者へ同じ点数を付けており相対的な差がないため、全レコードの `z` を0、`T`を50とする。
- 浮動小数点誤差により分散がごく小さい負数になった場合は0として扱う。
- 一人の応募者に対する評価者が二人未満の場合、または参加した評価者が局の全評価項目を入力していない場合、その応募者は「評価不足」とし補正後順位を確定しない。
- 補正得点は理論上0未満または100超になり得る。順位を歪めないため、上下限で丸めない。
- 補正後得点は派生データであるためDBへ保存せず、ランキング表示時に素点から再計算する。

SQLでは `COUNT(score)`、`AVG(score)`、`AVG(score * score)` を取得し、平方根以降の計算をPythonで行う。平方根は `variance ** 0.5` で計算できるため、新しい外部モジュールは不要である。

## 8. 主要な画面と受け入れ条件

### 8.1 入口

- システム概要を表示する。
- 応募者、面接官、局責任者の各メニューへ移動できる。
- 提出者の氏名・学籍番号を表示しない。

### 8.2 応募登録

- 氏名、メールアドレス、希望局、備考、複数の希望面接枠を入力できる。
- 希望局を選ぶと、その局の局別面接枠だけを選択肢として表示する。
- 必須値、長さ、制御文字、メール形式、局・枠の存在、局と枠の一致をサーバー側で検証する。
- 応募者と希望面接枠を一つのトランザクションで登録する。
- 途中で失敗した場合は、応募者だけが残らないようロールバックする。

### 8.3 応募者一覧・詳細

- 局、氏名、日程確定状況で絞り込める。
- 応募者詳細で、応募者情報、希望枠、確定枠、評価者、評価入力状況を確認できる。
- SQLの入力値は必ずプレースホルダーで渡す。

### 8.4 面接割り当て

- 局責任者の面接枠管理画面で、選択した局の応募者を未割り当て欄と面接枠欄に分けて表示する。
- 応募者カードをドラッグ＆ドロップして、希望している未予約枠へ手動で割り当てられる。
- 未割り当て欄へ戻すことで、評価入力前の確定枠を解除できる。
- 応募者が希望した枠以外、別の応募者が予約済みの枠は画面上でドロップできない。
- 確定時に、枠が未予約であることを再確認する。
- 同時更新でUNIQUE制約違反が発生した場合は、二重予約として利用者向けエラーを表示する。
- 割り当て変更はUPDATEで行う。

### 8.5 局別面接枠管理

- 既存の共通時間帯を選ぶか、開始・終了時刻を入力して、局責任者が局別面接枠をINSERTできる。
- 新しい時刻を入力した場合は、`schedules` と `bureau_schedules` を一つのトランザクションで登録する。
- 登録・変更前に、同じ局の既存枠と時間が重複しないことを `new_start < existing_end AND new_end > existing_start` で検証する。
- 未使用の局別面接枠はDELETEできる。
- 希望または確定面接で参照されている枠は、確認なしに削除できない。

### 8.6 評価入力

- 面接官を個別の面接枠へ割り当てる処理は設けない。
- 面接官メニューで面接官を選択すると、その面接官と同じ局の面接確定済み応募者を表示する。
- 面接官は、自分と同じ局の応募者だけを評価できる。
- 応募先局の全評価項目について1〜10の整数を入力する。
- 全項目を一つのトランザクションで登録・更新する。
- 一項目でも不正なら、どの項目も更新しない。
- 一人の応募者は、少なくとも二人の面接官から全評価項目の評価を受ける。

### 8.7 ランキング・採否

- 局を指定し、応募者ごとの補正後総合得点を高い順に表示する。
- 監査可能性のため、補正前の単純平均、補正後総合得点、評価者数も併記する。
- AVG、COUNT、GROUP BY、複数表JOINを使用して、評価者・評価項目別の統計量を取得する。
- 未評価、評価項目不足、評価者不足、補正用標本不足の応募者を、順位確定済み応募者と区別して表示する。
- 局の採用可能人数を表示する。
- 自動計算結果は「順位・採用候補」とし、最終的な `decision` の更新は局責任者の確定操作で行う。
- 同点が採用境界にある場合は自動確定せず、同点であることを表示する。

## 9. SQL・データモデリング評価項目との対応

- SELECT: 応募者一覧、詳細、空き枠、局別面接、ランキング
- JOIN: 応募者・局・局別面接枠・時間帯・面接官・評価項目・評価
- 集約関数: AVG、COUNTによる評価者別統計、補正後ランキング、評価進捗
- VIEW: `applicant_overview`
- INSERT: 応募者、希望枠、局別面接枠、評価
- UPDATE: 確定面接枠、評価、採否
- DELETE: 未使用の局別面接枠、不要になった希望枠
- トランザクション: 応募者＋希望枠の登録、希望枠の一括変更、全評価項目の保存
- 制約: PRIMARY KEY、FOREIGN KEY、UNIQUE、CHECK、NOT NULL

## 10. エラー処理・セキュリティ

- SQL文字列へユーザー入力を連結せず、すべてプレースホルダーを使用する。
- Jinja2の自動エスケープを無効化しない。
- 更新・削除はPOSTで行い、GETアクセスだけでDBを変更しない。
- URLのID、フォームのID、点数、日時、必須文字列をサーバー側で再検証する。
- `request.form[...]` の欠落による未処理例外を避け、利用者向けエラーを表示する。
- `sqlite3.IntegrityError` とその他の `sqlite3.Error` を区別し、失敗時はロールバックする。
- DB接続ごとに `PRAGMA foreign_keys = ON` を実行する。
- デバッグ表示を本番で有効にしない。
- 認証なしのため実在個人のデータを入れず、役割別メニューをアクセス制御と誤認させない。

## 11. 完了条件

- 全画面がローカル環境とWS室本番環境で動作する。
- スキーマ初期化手順と、元に戻せる初期DBコピーがある。
- 主要な正常系・異常系を手動または自動テストで確認する。
- 二重予約、局の不一致、1〜10外の評価、外部キー違反が保存されない。
- 使用した全SQL文を一覧化し、意味を最終レポートで説明できる。
- 第3課題からのスキーマ変更点と変更理由を最終レポートに記載する。
- 架空データだけを含むZIPと、本番配置中のファイルに差分がない。

## 12. 決定事項

### D-01 評価者ごとの点数補正

評価者×評価項目単位のz得点を使用し、第7.4節の数式と例外規則に従う。補正前の素点と補正後得点を両方表示し、計算過程を確認可能にする。

### D-02 認証と権限

講義資料に従って認証は実装せず、役割別メニューのみ作る。レポート課題3の「権限によって操作・閲覧を限定」は、最終課題では対象外へ変更したと明記する。

### D-03 日程割り当て方式

応募者の希望枠を見ながら局責任者が手動割り当てする。自動最適化は対象外とする。同じ局で同時刻に複数の面接は行わないため、局と共通時間帯の組で一つの面接枠が確定する。面接官と面接枠の割り当ては管理しない。

### D-04 採否決定

補正後総合得点順の採用候補を表示するが、自動で採否を保存しない。局責任者が確定し、採用境界の同点は手動判断とする。

### D-05 評価項目の重み

全評価項目を同じ重みとする。`criteria.weight` は追加しない。
